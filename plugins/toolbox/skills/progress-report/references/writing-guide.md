# progress-report 執筆ガイド（詳細）

本文（SKILL.md）の各ルールの詳細・例・矛盾処理。書きながら迷ったらここを読む。

## 情報源チェックリスト（Mode B では上から順に確認）

特定のツール名には依存しない。実行環境にあるコマンド・ファイル閲覧手段で確認する。

1. **上位計画・タスク体系**: 計画ファイル（`~/.claude/plans/`、`docs/plan*`、`PLAN.md`）、エピック/親Issue、マイルストーン定義、ロードマップ。**本作業が属する「大元のゴール」を最初に特定する**
2. **バージョン管理の履歴**: コミットログ（`git log --oneline --stat -30` 等）、ベースブランチとの差分、未コミット変更、stash
3. **タスク管理**: TODO ファイル、チェックリスト、Issue / PR とそのコメント
4. **実行結果**: テスト・ビルド・lint の状態（実行可能なら実際に走らせる）、CI の状態
5. **作業ログ**: セッションログ、エージェントの作業メモ・レポートファイル

### どこまで掘るか

次の4つに答えられたら止めてよい: (1) **大元のゴールは何で、本作業はその何番目のピースか** (2) この作業は何を達成しようとしているか (3) どこまで終わって何が残っているか (4) 順調か、壊れて/詰まっているか。

「うまくいっていない兆候」（失敗テスト・revert・大量 fixup・中途半端な未コミット）を見つけたら浅く流さず、「なぜ今壊れたのか」（どの変更の波及か）まで遡る。読み手が最も知りたいのは順調な部分ではなく問題と判断が必要な箇所。

## 事実と推測の区別（Mode B では必須）

推測を断定口調で書いた報告は、間違った判断を誘発するため報告しないより有害。

- 痕跡から直接確認できたことは事実として書く（「テスト 3 件が失敗している」）
- 推測は推測と明示し根拠を添える（「認証方式の変更が原因と**推測される**（根拠: 失敗テストは全て auth モジュール、直前コミットで token 検証を変更）」）
- 確認できなかったことは「不明」と正直に書く。セクションを埋めるための創作をしない
- **痕跡同士が矛盾する場合**は、より新しく・実行可能な痕跡（テスト・コード）を意図の根拠として優先し、「計画書と実装が食い違っており、〜を正とみなした」と矛盾の事実と採った解釈を一行残す（読み手が解釈を覆せるように）。痕跡が挙動を部分的にしか定義していない場合も、定義済み部分は事実、未定義部分は「推奨つきの確認事項」として読み手に回す

## 執筆ルール詳細

- **同一事項が複数セクションに該当する場合**（例: スコープ外だが人間対応が必要なリスク）: 各セクションにはそのセクションの役割の側面だけを書く — §2 は「何をしてほしいか」、§3 は「対象範囲の事実」、§6 は「in-scope の作業」、§7 は「リスク・経緯」。詳細は最も役割が合う1箇所に置き、他からは一行+参照で繋ぐ
- **数字の検算**: 内訳の合計が総数と一致するか検算する（成功+失敗+skip=総数。skip を成功に数えない）。テストランナーの failure/error/skip の区分は丸めない
- **専門用語は初出時に一言補足**（「PASETO（JWT の代替トークン形式）」程度）
- **メタ情報を混ぜない**: 保存先パス・使用ツール・前任者のローカルパス等、読み手の判断に関係しない舞台裏は報告の外（呼び出し元への返答側）へ
- **急ぎ度の書き方**: 高/中/低のラベル単独にせず「それが解消されないと何がいつから止まるか」を必ず添える
- **ステータス色**: 🔴=ブロッカーまたは読み手の判断待ちあり、🟡=リスクはあるが進める、🟢=どちらもない。迷ったら悪い方の色

## Good / Bad 例

同じ状況（決済モジュールのリファクタリング中、テスト2件失敗、レート制限の仕様判断が必要）の対比。

**Bad** — 問題が最後尾・網羅列挙・問題隠し・上位文脈なし:

```markdown
# 進捗報告
## やったこと
- payment.ts を修正
- payment_test.ts を修正
- utils.ts の import を整理
-（…以下12項目続く）
## その他
特に問題ありません。順調です。
```

→ 大元の何のどこをやっているか分からない。何が重要か分からない。失敗テストに触れず「順調」は事実に反する。依頼が存在しない。

**Good** — 結論先行・上位ゴールへのアンカー・依頼明示・問題とアクションの接続:

```markdown
# 進捗報告: 決済リトライ機構の実装（2026-06-06）

**ステータス**: 🔴 要判断 — 決済安定化イニシアチブ 2/3 タスク目・実装完了・テスト 40/42（2件は仕様未定義で要判断）

## 1. TL;DR
- 大元は「決済安定化イニシアチブ」（全3タスク）。本作業はその2つ目「リトライ機構」
- 実装完了・テスト 42 件中 40 件成功。残り2件はレート制限時（HTTP 429）の挙動が仕様未定義で失敗中
- **判断のお願いが1件**（→ §2）

## 2. 要対応・要判断
1. **HTTP 429 でリトライすべきか即時エラーにすべきか決めてください。**
   外部 API の課金体系（リトライが課金対象か）に依存するため実装側では判断できません。
   決まればテスト2件を確定でき本件完了。返答があるまで次タスク（3つ目: 監視強化）へ先行着手します。

## 3. 全体マップ
| # | タスク | 状態 | 検証レベル |
|---|---|---|---|
| 1 | タイムアウト調整 | ✅ 完了 | 本番反映済み |
| 2 | **リトライ機構（本作業）** | 🔄 仕様判断待ち | ローカルテストまで |
| 3 | 決済監視の強化 | ⬜ 未着手 | — |
```
