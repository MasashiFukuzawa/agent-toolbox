# 認知負荷の一般則 — html-thinking の視覚優先ルールの根拠

SKILL.md の定量ルール（3項目→表、関係→図、文字量上限）の背景。ルールの妥当性を疑うとき・例外を判断するときに読む。

## なぜ「文字より図表」か

- **走査 vs 精読**: 画面の読み手はまず F パターンで走査し、読む価値を判断してから精読する。散文の壁は走査段階で構造を渡せず「読む価値の判断」自体を妨げる。表・図は走査段階で構造（何がいくつあり、どう関係するか）を渡せる
- **チャンク限界**: 作業記憶が同時に保持できるのは約4±1チャンク。7項目の箇条書きは読者に「チャンク化の作業」を丸投げしている。表は列がチャンク化を代行し、図は空間配置が関係の記憶を代行する
- **二重符号化**: 空間・視覚情報は言語と別チャネルで処理される。同じ内容でも図+短文は文章単独より再生率が高い

## 1画面1メッセージ

- 1つのビューポート（スクロールなしで見える範囲）が伝えてよい主張は1つ。複数の主張を1画面に詰めると、どれも記憶に残らない
- 「概要 → セクション → 詳細」の階層は、各層で1メッセージを守るための構造。`<details>` はこの階層をページ内で実現する装置

## 例外（散文が正しい場合）

- **因果の説明・トレードオフの叙述・注意喚起**: 構造化すると論理の流れが切れるもの。ただし1ブロック3文まで、必ず対応する図表に隣接させる
- **1〜2項目しかない情報**: 表にすると空疎になる。文で書く
- 図解が「デコレーション」になる場合（構造がない情報を無理に箱と矢印にする）はやらない — 図は関係が存在するときだけ意味を持つ

## セルフチェックへの翻訳

1. 初見5秒で「このページの答え」が言えるか（key question 契約）
2. 1画面に主張がいくつあるか（2つ以上なら分割か降格）
3. 箇条書きで3項目以上並んでいる場所はないか（→表）
4. 「AはBに依存し、CはAとBの後…」のような関係の散文はないか（→図）
