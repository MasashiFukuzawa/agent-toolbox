# Test Smells And Refactor Recipes

## 仕様根拠が弱い

兆候:

- 期待値が実装の定数や分岐のコピーになっている。
- 「現在こう動くから」を唯一の根拠にしている。
- テスト名がメソッド名の言い換えで、ドメイン上の事実を表していない。

対処:

- 仕様オラクルを探す。
- 仕様がないならユーザーに確認する。
- 現在挙動を固定する必要がある時だけ characterization test と明記する。

## 同義反復assert

兆候:

- プロダクションコードの関数を使って期待値を計算している。
- プロダクションコードの定数をそのまま期待値に使っている。
- 実質的に「実装が実装と等しい」ことだけを確認している。

対処:

- 期待値はテスト内に独立して書く。
- ドメイン用語で期待される結果を表現する。
- 複雑な期待値は、小さな代表ケースや境界ケースへ分ける。

## 過剰モック

兆候:

- 同一プロセス内のサービス呼び出しを検証している。
- 内部Repository呼び出しを検証している。
- スタブへの呼び出し回数を検証している。
- 具象クラスやサードパーティSDKを直接モックしている。

対処:

- 公開振る舞い、戻り値、公開状態、永続化後の状態で検証する。
- 管理下にない依存だけを所有アダプタ経由でモック/スパイ化する。
- 具象クラスを部分モックしたい場合は責務分離を検討する。

## private method の直接テスト

兆候:

- private を public/internal に変える提案が出る。
- reflection や言語の抜け道で private を呼ぼうとしている。

対処:

- 公開振る舞いから間接的に検証する。
- 公開振る舞いから検証できないほど重要なら、抽象化が不足している。別クラス、値オブジェクト、関数的核へ抽出する。

## プロダクションコード汚染

兆候:

- `isTest`、test-only setter、test-only constructor を本体に追加する。
- テストだけのためにカプセル化を壊す。
- 現在日時をグローバルなambient contextで差し替える。

対処:

- 依存はコンストラクタまたは引数で明示する。
- 時刻は可能な限り値として渡す。
- テスト容易性のための変更は、プロダクト設計としても説明できる最小変更にする。

## snapshot濫用

兆候:

- 大きなDOMやJSON全体をsnapshotに保存している。
- 何が仕様なのかテスト名やassertから読めない。
- 小さなUI構造変更で頻繁に壊れる。

対処:

- ユーザーが観察する文言、状態、アクセシビリティ、主要な契約だけをassertする。
- snapshotを使う場合は、差分が仕様判断に役立つ小さな単位にする。
- 広いsnapshotは承認テストとして目的を明記する。

## flaky test

兆候:

- `sleep` や固定待ち時間に依存する。
- 現在時刻、乱数、UUID、実行順序に依存する。
- 並行処理の順序を暗黙に仮定する。
- 外部ネットワークや共有DB状態に依存する。

対処:

- 明示的な同期ポイント、event、polling with timeoutを使う。
- clock、random seed、ID生成を制御する。
- 並行性は結果の不変条件を検証し、順序を仕様にしない。
- テストごとに独立したデータを作り、開始時に掃除する。

## カバレッジ稼ぎ

兆候:

- assertが弱い、または存在しない。
- trivial getter/setter、単純委譲、フレームワーク配線だけをテストしている。
- カバレッジを上げるために分岐をなぞっているが、ドメイン上の失敗を検出しない。

対処:

- 守る価値のある振る舞いへ対象を移す。
- 取るに足らないコードはテストしない。
- 複雑さやドメイン重要性が高い部分に集中する。

## ログのテスト

診断ログは開発者向けの実装詳細なので、原則テストしない。サポートログは運用者やサポート担当者が使う仕様上の出力であれば、観察可能な振る舞いとしてテストしてよい。

ログテストでも、ログライブラリを直接モックしない。必要なら所有するログ境界やドメインイベントを検証する。
