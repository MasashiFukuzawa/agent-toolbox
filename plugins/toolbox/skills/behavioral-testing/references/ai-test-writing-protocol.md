# AI Test Writing Protocol

## 1. Repo探索

テスト実装前に次を確認する。

- 変更対象またはテスト対象の公開API。
- 仕様根拠: issue、PRD、README、API docs、型、既存テスト、バグ再現手順。
- テストランナーと最小実行コマンド。
- 既存テストの配置、命名、AAA/Given-When-Then、assertion style。
- fixture、factory、builder、object mother、seed data。
- DBや外部依存のテスト方針。
- 時刻、乱数、UUID、並行処理、非同期処理の扱い。
- CIで実行されるコマンドとローカルで実行可能な最小コマンド。

`rg`、`rg --files`、manifest、CI設定、既存テストを優先して読む。既存規約がある場合は、それを破らない。

## 2. 仕様オラクル

期待値は次のいずれかから作る。

- ユーザーが明示した仕様。
- issue/PRD/API契約。
- バグ再現条件と修正後の望ましい挙動。
- 既存テストで表現された明示的な契約。
- ドメイン用語や公開型から推論できる不変条件。

期待値を現在の実装からだけ作らない。実装の分岐、定数、アルゴリズムをテスト側にコピーすると、同じバグを二重化するだけになる。

仕様が不明な場合の分岐:

- ユーザーに確認できるなら確認する。
- 現在挙動を守る目的が明確なら characterization test と明記する。
- 仕様のない新規期待値を作る必要があるなら、テスト実装を止める。

境界仕様が不明な場合:

- 境界値テストの期待値をAIが補わない。
- 包含/排他、丸め、時刻の基準、timezone、同値時の扱いは仕様または公開契約で確認する。
- 仕様化済みの代表ケースだけを実装し、未確定の境界は質問または保留メモにする。
- characterization test として固定する場合も、正しい仕様ではなく現状記録であることを明記する。

## 3. テスト提案フォーマット

大きめの変更や曖昧な依頼では、実装前に短く整理する。

```text
仕様根拠:
観察する振る舞い:
テスト対象分類:
テストレベル:
依存の扱い:
使う既存fixture/factory:
mock/spyを使う理由:
時刻/乱数/並行性/外部I/Oの扱い:
実行コマンド:
```

単純な修正ではこの形式をユーザーに長く出さなくてよいが、内部判断としては埋める。

## 4. テスト実装

- 1テストは1つの振る舞いを守る。
- Arrange は必要最小限にする。意図が読めるfactory/helperを使う。
- Act は原則1つの操作にする。
- Assert は外部から観察可能な結果だけを見る。
- テスト名は「状況と期待される振る舞い」を表す。
- 既存プロジェクトの言語、フレームワーク、命名規約を優先する。

避けること:

- private method の直接テスト。
- テストのためだけの public 化。
- 本体コードへの `if test`、test-only hook、ambient context の追加。
- カバレッジのためだけのassert。
- snapshotだけで意味を説明しないテスト。
- `sleep` に依存する非同期テスト。
- 実ネットワーク、実メール、実決済への通信。

## 5. データ設計

- ビジネス的に重要な値はテスト本文から読めるようにする。
- 技術的な準備はfactory/helperへ寄せる。
- ランダム値は原則固定する。ランダム化が目的ならseedを固定し、失敗時に再現できるようにする。
- 現在時刻は値として注入できるなら値で固定する。サービス注入が必要なら既存パターンに合わせる。
- 境界値、代表値、異常値を選ぶ。網羅率だけのために無意味なケースを増やさない。
- 境界値は、期待値の仕様根拠がある場合だけ追加する。境界仕様が未確定なら、テストケースではなく確認事項として扱う。

## 6. 検証

優先順:

1. 追加・変更したテストだけを実行する。
2. 影響するテストファイルまたはテストグループを実行する。
3. 必要に応じて全体テストやCI相当コマンドを実行する。

失敗した場合:

- テスト期待値が仕様に合っているか確認する。
- 実装バグか、テストの前提ミスか、環境問題かを分ける。
- テストを通すために仕様を下げない。
- 本体コードを変える場合、テスト都合ではなくプロダクト上の設計改善として説明できることを確認する。

## 7. 報告

最終報告には次を含める。

- 追加・変更したテストが守る振る舞い。
- 期待値の根拠。
- モック/スパイを使った場合はその理由。
- 実行した検証コマンドと結果。
- 実行できなかった検証があれば理由。
